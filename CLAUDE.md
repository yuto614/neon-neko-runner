# CLAUDE.md

このファイルは、このリポジトリで Claude Code が作業する際のルールをまとめたものです。

## プロジェクト概要

「猫」をモチーフにした、ブラウザで遊べるアクション/ミニゲーム。

障害物を避けたり、ジャンプしたりするシンプルな操作で、誰でもすぐに遊べることを目指す。

具体的なゲームルールは今後詰めていく（このファイルの「ゲーム設計」セクションを更新していく）。

## 技術スタック

* プレーンな HTML / CSS / JavaScript（ES Modules）のみ
* 描画は Canvas API を使用する
* フレームワーク・ビルドツール（React, Vue, Vite, webpack など）は導入しない
* TypeScript は導入しない
* npm 等のパッケージ管理は基本的に使わない
* 外部ライブラリは原則追加しない
* CSS フレームワーク（Bootstrap、Tailwind CSS など）は導入しない

技術スタックを変更したい場合は、必ず事前にユーザーへ確認すること。

## ディレクトリ構成（方針）

```text
index.html      # エントリーポイント
css/            # スタイル
js/             # ゲームロジック（ES Modules）
assets/         # 画像・音声などのリソース
```

## コーディング規約

* JavaScript は ES2020+ を使用する
* `const` と `let` のみを使用し、`var` は使用しない
* インデントはスペース2つ
* 1ファイル1責務を意識する
* グローバル変数を増やさない
* コメントは必要最小限とし、「なぜそうしているか」を説明する場合のみ記述する
* 「何をしているか」を説明するコメントは書かない
* 今必要な分だけ実装する
* 過剰な抽象化や将来のためだけの拡張は行わない

## 実装方針

* まず動く最小構成（MVP）を優先する
* 一度に大量の機能を追加しない
* 小さな単位で実装し、動作確認しながら進める
* 要件が曖昧な場合は勝手に決めず、ユーザーに確認する
* 推測で機能を追加しない
* ユーザーが要求していない機能を追加しない

## ファイル作成ルール

* 新しいファイルやディレクトリを作成する前に、本当に必要か確認する
* 同じ役割のファイルを増やさず、既存ファイルへの追加を優先する
* 大規模なリファクタリングやディレクトリ構成変更は事前にユーザーへ確認する

## Canvas実装方針

* `requestAnimationFrame()` を用いて描画する
* 毎フレーム `update()` → `render()` の順で処理する
* FPSを固定しない
* 不必要なクラス化は行わない
* 必要最小限のモジュール分割を行う

## CSS方針

* プレーンな CSS を使用する
* インラインスタイルは原則使用しない
* 必要以上に複雑なセレクタを書かない
* 必要に応じて CSS変数を使用する
* 見た目の調整だけのために JavaScript を増やさない

## JavaScript方針

* DOM操作は標準 API のみを使用する
* 不要なクラス化を行わない
* 必要になるまでデザインパターンを導入しない
* 未使用の関数や変数を残さない
* 非同期処理は必要最小限にする

## UI方針

* シンプルで見やすいデザインを優先する
* 派手なアニメーションや過剰な装飾は追加しない
* PCとモバイルの両方で遊べるようにする
* 色やレイアウトを大きく変更する場合は事前にユーザーへ確認する

## アセット

* 画像や音声は assets/ 以下に配置する
* 仮素材は後から差し替えやすくする
* 外部素材を追加する場合は事前にユーザーへ確認する
* AI生成画像やAI生成音声を使用する場合は事前にユーザーへ確認する
* ライセンスが不明な素材は使用しない

## ゲーム設計（初期方針）

* ジャンル: 障害物を避ける / ジャンプするアクション系ミニゲーム
* キーボード（スペースキーや矢印キー）に加え、クリックやタップでも操作可能にする
* モバイル端末でも遊べるようにする
* スコアとハイスコアは `localStorage` に保存する
* 詳細なルールやキャラクターデザインは実装を進めながら決定する

## セーブデータ

* 永続化は `localStorage` を優先する
* IndexedDB やサーバー保存は必要になった段階で検討する
* セーブデータ形式を大きく変更する場合は事前にユーザーへ確認する

## パフォーマンス

* まず正しく動くことを優先する
* 必要以上の最適化を行わない
* ボトルネックが確認できた場合のみ最適化する
* 不必要なキャッシュ機構を導入しない

## デバッグ

* エラーを隠すための try-catch を安易に追加しない
* console.log の追加は最小限にする
* 問題を修正する際は原因も考慮する
* 一時的なデバッグコードは残さない

## リファクタリング

* 動作しているコードを大規模に書き換えない
* 必要性がある場合のみリファクタリングする
* 可読性向上のみを目的とした大規模変更は避ける
* リファクタリング後も既存の挙動を維持する

## セキュリティ

* 外部 API や外部サーバーへの通信を勝手に追加しない
* ユーザー情報を収集しない
* Cookie を利用する場合は事前にユーザーへ確認する

## 依存関係

* npm install はユーザーの許可なしに実行しない
* package.json を作成する場合は事前にユーザーへ確認する
* 外部ライブラリを追加する場合は理由を説明したうえで確認する

## 動作確認方法

* ビルド不要
* `index.html` を直接開くか、Live Server 等のローカルサーバーで確認する
* UIや見た目を変更した場合は、実際にブラウザで動作確認する

## デプロイ

* 静的ファイルのみで構成する
* GitHub Pages 等の静的ホスティングを想定する
* デプロイ作業は事前にユーザーへ確認する

## Git

* commit や push はユーザーの指示があった場合のみ行う
* force push は行わない
* branch の作成や切り替えは事前にユーザーへ確認する
* `.gitignore` の変更は事前に確認する

## Claude Code の行動原則

* シンプルさを最優先する
* コード量よりも分かりやすさを優先する
* 完璧さよりも保守しやすさを優先する
* 不必要な抽象化を行わない
* 将来のためだけのコードを書かない
* 必要になるまでデザインパターンを導入しない
* 推測で実装しない
* 要件に複数の解釈がある場合はユーザーに質問する
* 不明点を勝手に補完しない

以下のような仕組みは、本当に必要になるまでは導入しない。

* SceneManager
* EventBus
* AssetLoader
* ECS（Entity Component System）
* GameObject の基底クラス
* 独自フレームワーク
* 過度な継承構造

## コミュニケーション

* ユーザーとのやり取りは日本語で行う
* 返信は簡潔にする
* 差分やコードを見れば分かることを文章で繰り返さない
* 実装前に簡単な方針を示す
* 完了後は変更点のみ簡潔に報告する

## やらないこと

* ユーザーの許可なくフレームワークやビルドツールを導入しない
* ユーザーの許可なく TypeScript を導入しない
* ユーザーの許可なく外部ライブラリを追加しない
* ユーザーの許可なく依存関係を変更しない
* ユーザーの許可なく破壊的な変更を行わない
* ユーザーが要求していない機能を勝手に追加しない
* force push を行わない
